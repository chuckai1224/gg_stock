## 核心論點拆解

### 1. 為什麼想離開 Claude Code（痛點）

- **慢**：台灣晚上（歐美上班時間）尖峰時段回應明顯變慢，重度使用者一天來回 10–20 次受不了。
- **貴**：新模型 Fable 很聰明但 Token 消耗極快，官方送額度也不夠燒。
- **變笨（體感）**：懷疑 Opus 在 Fable 上線後變弱，或使用人數變多導致品質下降。

同時給 Codex 的正面評價：改檔案、跑程式、測試網站、改小功能**速度快很多**；做圖片、PDF、簡報效果不錯（內建生圖不用另外花 API 錢）。

### 2. 真正的問題：AI 資產搬遷，不是工具安裝

累積的四類資產：
1. 全域規則（個人偏好、工作原則）
2. 各專案規則（啟動方式、資料庫位置、禁碰檔案）
3. 調教多時的 Skills（教給 AI 的工作手冊）
4. 長期累積的記憶與工作流程

關鍵障礙：Claude Code 只認 `CLAUDE.md`，Codex 只認 `AGENTS.md`——換工具後 AI「一個字都看不到」你的積累，看起來就像變笨。

### 3. 反面教材：複製兩份 = 維護地獄

- 一個月後在 A 邊加規則忘了同步 B 邊
- 三個月後 skills 版本分裂，搞不清哪份是正本
- 未來每多一個工具（如 Grok 的 coding agent）就要再複製第三份

### 4. 解法：一份正本、多個入口

```
~/                          # 使用者目錄（全域層）
├── .ai/
│   ├── INSTRUCTIONS.md     # 全域規則唯一正本
│   └── skills/             # 共用 skills 唯一正本
├── CLAUDE.md   → symlink → .ai 正本
└── AGENTS.md   → symlink → .ai 正本

project/                    # 專案層（同一套邏輯）
├── .ai/
│   └── INSTRUCTIONS.md     # 專案規則唯一正本
├── CLAUDE.md   → symlink
└── AGENTS.md   → symlink
```

設計原則：
- **正本命名中立**（`INSTRUCTIONS.md`）：正本「不姓 Claude 也不姓 OpenAI」，未來新 AI 只要再加一道門（symlink）。
- **全域 vs 專案嚴格分層**：全域放跨專案通用原則（先讀專案規則、密碼不寫死、不破壞軟連結…）；專案細節（怎麼啟動、DB 在哪、測試怎麼跑）留在專案內。
- **Skills 分兩類**：能跨工具共用的才進 `.ai/skills/` 共用區；依賴平台專屬功能的（如 Codex 內建生圖、Claude 專屬呼叫）留在各自平台，「共同知識不重複，不同能力各司其職」。

### 5. Claude Code vs Codex：分工而非競爭

| | Claude Code | Codex |
|---|---|---|
| 擅長 | 梳理長篇規格、架構推理、風險分析、審查驗證（尤其 Fable 的規劃與驗證能力） | 執行速度：改程式、跑指令、測試、操作瀏覽器、桌機/手機版檢查、生圖、PDF、簡報、Google Sheet |
| 弱點 | 慢、燒 Token、不能自己生圖 | 寫計劃書和驗證時比較草率 |
| 角色 | 建築師（畫設計圖 + 最終驗收） | 施工團隊（一路做完交成果） |

推薦工作流：**Claude Code 規劃 + 定驗收條件 → Codex 實作 + 測試 → 丟回 Claude Code 最終審查**。附帶省錢案例：以前生圖要走 OpenRouter 呼叫 API，現在 Codex 內建生圖「一毛 API 都不用花」。

### 6. 核心心法

> 「絕對不要忠於任何一個 AI 工具，你該忠誠的是你自己的效率。」

工具只是入口，資產永遠在自己手上；CP 值一降就走人，哪天 Claude Code 變強再切回來。

## 商業模式觀察

影片後段（11:17–13:50，約 3.5 分鐘）是導流段，鉤子是「一鍵搬遷提示詞 + PDF 教學」，放在付費社群「AI 投資學院 Plus」（Skool）內，並展示社群內容：Claude Code 課程、AI+XQ 程式交易密訓班、n8n 自動化大師班、AI 投資資訊站（台/港/美股董事吃貨、市場雷達、匯出 Excel）、AI 飆股雷達實例（6488 環球晶 5/20 由虧轉盈訊號、6/19 強烈買進、一個月翻倍）。

典型的「免費講方法論 → 完整工具包放付費社群」結構，教學內容本身完整可自行實作，付費點在於省事（現成提示詞/模板）。

## 評價

**優點**
- 方法論扎實且通用：single source of truth + symlink 入口，是軟體工程正統做法（DRY），不只適用 AI 設定。
- 「全域/專案分層」「共用/專屬 skills 分流」兩個細節講得清楚，是多數同類影片會漏掉的坑。
- 「忠於效率不忠於工具」的框架有記憶點，分工建議（Claude 想、Codex 做）符合兩工具的實際特性。

**可以更完整的地方**
- 未提及 Windows 上 symlink 的坑（`mklink` 需系統管理員權限或開發者模式；git 對 symlink 的處理；某些工具不追蹤 symlink）。
- 「記憶檔案」的搬遷實際只講了規則與 skills，記憶（memory）如何共用未展開。
- Claude Code 的 hooks、settings.json、MCP 設定等無法用 symlink 共用的部分沒有討論。
- 「Opus 變笨」屬體感描述，無測試數據支撐（影片自己也承認是體感）。

## 三個重點總結（影片原話）

1. **規則只有一份**：中立正本放 `.ai/`，`CLAUDE.md`、`AGENTS.md` 都只是入口。
2. **各司其職**：Claude Code 負責想（規劃/驗收），Codex 負責做（實作/測試）。
3. **忠於效率，不忠於工具**：資產在自己手上，隨時可以跳槽。
